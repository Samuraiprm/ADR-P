# Этап 3: Реагирование и визуализация

## Цель
Замкнуть цикл OODA: от вердикта до действия и обратной связи через дашборд.

## Функциональные требования
1. **Response Service (Go):**
   - Чтение из `abuse:verdicts`.
   - Для `BLOCK`: вызов API бана (мок или реальный TG/VK).
   - Для `WARN`: отправка алерта в Telegram с кнопками "Подтвердить / Отклонить".
   - Callback-хендлер для кнопок: обновление статуса в БД.
2. **Dashboard (Grafana):**
   - Панель "Real-time": RPS, распределение вердиктов, топ user_id.
   - Панель "Rules Efficiency": количество срабатываний по каждому правилу.
   - Панель "ML Performance": distribution of anomaly scores.
3. **API для управления:**
   - `GET /api/v1/rules` — список правил.
   - `POST /api/v1/rules` — создание нового правила.
   - `GET /api/v1/stats?from=&to=` — агрегированная статистика.

## Acceptance Criteria
- [ ] Кнопка в Telegram обновляет запись в БД в течение 2 секунд.
- [ ] Дашборд отображает данные с лагом < 30 секунд.
- [ ] API защищён базовой авторизацией или JWT.
- [ ] README содержит скриншоты дашборда и примеры curl-запросов.
